# Workflow — Local Setup, Deployment, CI Pipeline

This file covers everything about **moving plugin code** between environments:
workstation → local Docker Moodle → staging (Hetzner `dev.lernhive.de`) →
Plugins Directory. It is Moodle-5.x-aware and assumes macOS + OrbStack as the
local dev stack. The reference project is LernHive (`plugins/` layout), but
every pattern generalizes to any Moodle multi-plugin workspace.

---

## 1. Local dev stack

### Required tools

| Tool | Purpose | Install |
|---|---|---|
| **OrbStack** | macOS Docker/VM host (faster than Docker Desktop) | `brew install orbstack` |
| **Docker** | Container engine inside Orb | ships with OrbStack |
| **Moodle Docker** | Official Moodle docker-compose stack | `git clone https://github.com/moodlehq/moodle-docker` |
| **git / gh** | Version control | standard |
| **php 8.2+** | For running local tools (`vendor/bin/phpcs`, etc.) | `brew install php` |
| **composer** | Moodle dev dependencies | `brew install composer` |
| **node + npm** | For Grunt, AMD build, SCSS | `brew install node` |

### Local Moodle via Moodle Docker

```bash
# One-time setup
export MOODLE_DOCKER_WWWROOT=~/dev/moodle            # host-side Moodle git checkout
export MOODLE_DOCKER_DB=pgsql                        # pgsql | mariadb | mysqli | mssql
export MOODLE_DOCKER_PHP_VERSION=8.2

cd ~/dev/moodle-docker
cp config.docker-template.php "$MOODLE_DOCKER_WWWROOT/config.php"
bin/moodle-docker-compose up -d
bin/moodle-docker-wait-for-db
bin/moodle-docker-compose exec webserver php admin/cli/install_database.php \
    --agree-license --fullname="Dev" --shortname="dev" \
    --adminpass=password --adminemail=dev@example.com
```

After this, Moodle is reachable at `http://localhost:8000`.

### Moodle 5.x `public/` split

Starting with Moodle 5.0, the webroot is physically split:

```
<repo-root>/
├── composer.json          # project tools, vendors, phpunit.xml live HERE
├── vendor/                # PHPUnit, Behat, PHPCS binaries
├── phpunit.xml            # generated by Moodle's init.php, references paths
│                          # relative to repo root
├── public/                # <-- webroot ($CFG->wwwroot served from here)
│   ├── index.php
│   ├── config.php
│   ├── lib/
│   ├── mod/
│   ├── local/             # install local plugins here (NOT at repo root)
│   ├── theme/
│   └── admin/cli/...
└── tests/ behat.yml ...
```

**Key rule:** the "Moodle CLI root" (`public/`, where `admin/cli/upgrade.php`
lives) is **different** from the "Moodle repo root" (one level up, where
`composer.json`, `vendor/`, and `phpunit.xml` live). Tests need to run
**from the repo root**, deployment targets need to copy **into public/**.
Confusing these breaks tests silently.

Throughout this framework:
- `$MOODLE_CLI_ROOT` = webroot (`…/public/` on 5.x, `…/` on 4.x flat)
- `$MOODLE_REPO_ROOT` = composer.json+vendor+phpunit.xml location
- For 4.x classic flat layout, the two are equal.

## 2. Workspace layout for multi-plugin projects

The LernHive repo layout (one writable workspace → many Moodle plugins):

```
lernhive.de/
├── AGENTS.md                    # project-level DevFlow rules
├── CONTRIBUTING.md
├── README.md
├── plugins/
│   ├── local_lernhive/           # base plugin
│   ├── local_lernhive_contenthub/
│   ├── local_lernhive_copy/
│   ├── …
│   └── theme_lernhive/
├── playbooks/                    # provision/deploy/test scripts
│   ├── provision.sh
│   ├── deploy.sh
│   ├── test.sh
│   ├── server-test.sh
│   ├── test.local.env
│   └── test.hetzner.env
├── .github/
│   ├── workflows/
│   │   ├── deploy-hetzner.yml
│   │   └── test-hetzner.yml
│   └── scripts/
│       └── publish-test-result.sh
└── product/                      # cross-plugin product docs
```

**Key rule — plugin dirname = frankenstyle component name.**
A workspace dir named `local_lernhive_contenthub/` deploys to
`public/local/lernhive_contenthub/`, producing the frankenstyle component
`local_lernhive_contenthub`. The PHPUnit testsuite Moodle registers for it
is `local_lernhive_contenthub_testsuite`. The Behat tag is
`@local_lernhive_contenthub`. This 1:1 mapping is what makes automation
like `playbooks/test.sh`'s default filter work. Do not break it by renaming
plugin directories.

## 3. Deployment patterns

There are **three environments** in the LernHive project and each has its
own deploy path:

### 3a. Local Docker (OrbStack)

Use `playbooks/deploy.sh` (local variant) or the `moodle-deploy` companion
skill. Key mechanics:

1. Detect Orb VM (`orb list`) and the running Moodle Docker container
   (`docker ps`).
2. `docker cp` each `plugins/<name>/` directory to its destination inside
   `public/local/<stripped-name>/` (strip the `local_` prefix) or the
   appropriate `mod/`, `blocks/`, `theme/` path.
3. Run `php admin/cli/upgrade.php --non-interactive` inside the container.
4. Run `php admin/cli/purge_caches.php` inside the container.
5. **Gotcha: OPcache.** If `opcache.validate_timestamps=0` (default in
   production-like Docker images), edits to PHP files after `docker cp`
   are invisible until PHP-FPM restarts. Either restart the webserver
   container after deploy, or set `opcache.revalidate_freq=0` during dev.

### 3b. Staging: dev.lernhive.de on Hetzner

Deployed **automatically by GitHub Actions** on every push to `main`.
Workflow file: `.github/workflows/deploy-hetzner.yml`. Mechanics:

1. Workflow runs `rsync` over SSH to the Hetzner server.
2. Target path is inside a Docker container: `lernhive-webserver-1`,
   webroot `/var/www/html/public/`. Local plugins land in
   `/var/www/html/public/local/<stripped-name>/`.
3. After sync, the workflow runs `docker exec lernhive-webserver-1 sudo -u
   www-data php admin/cli/upgrade.php --non-interactive` and purges caches.
4. Secrets are on the GitHub side: `HETZNER_HOST`, `HETZNER_USER`,
   `HETZNER_SSH_KEY`.

Never `rsync` directly from a dev machine to Hetzner unless the pipeline
is broken — always go through `main`, so the server state is reproducible
from git history.

### 3c. Moodle Plugins Directory

See `references/04-submission.md`. Never the day-to-day dev path, but the
endpoint for any plugin that eLeDia wants to publish publicly.

## 4. `playbooks/test.sh` — test driver

Single script used both locally (target `local`) and on Hetzner
(target `hetzner`). Wraps PHPUnit + Behat, injects env from a per-target
`.env` file (e.g. `test.hetzner.env`):

```bash
CONTAINER=lernhive-webserver-1
MOODLE_CLI_ROOT=/var/www/html/public
MOODLE_REPO_ROOT=/var/www/html
PHPUNIT_BIN=/var/www/html/vendor/bin/phpunit
BEHAT_BIN=/var/www/html/vendor/bin/behat
PHPUNIT_MEM=512M
```

Usage:

```bash
# Full default run (only LernHive plugin testsuites, NOT Moodle core)
./playbooks/test.sh --target=hetzner

# Just PHPUnit, only one plugin
./playbooks/test.sh --target=hetzner --suite=phpunit --component=local_lernhive_contenthub

# Just Behat with an ad-hoc tag expression
./playbooks/test.sh --target=hetzner --suite=behat --tags=@javascript

# Force reinitialize test env (rebuild test DB, clean dataroot)
./playbooks/test.sh --target=hetzner --reinit

# Opt-in: run Moodle core tests too (~29k tests, slow)
./playbooks/test.sh --target=hetzner --full
```

**Default filter rule:** when neither `--component` nor `--full` is given,
`run_phpunit()` enumerates `plugins/*/tests/` and passes the matching
`<dirname>_testsuite` names (comma-separated) to `phpunit --testsuite`.
This prevents the classic mistake of running ~29k Moodle core tests on
every CI run. The same pattern is mirrored in `run_behat()` via an
OR-expression of `@<plugin>` tags.

**Warnings-vs-failure parsing:** PHPUnit 10 with `failOnWarning` treats
even harmless `reset_dataroot` teardown warnings as exit-non-zero. The
test driver therefore captures output, inspects the last lines, and
treats `^OK (N tests, M assertions)` or `^OK, but there were issues!`
as **pass** — only `FAILURES!` / `ERRORS!` map to a real test failure.
Do not remove this parsing without also fixing Moodle core's teardown.

## 5. CI pipeline (GitHub Actions)

Two workflows in `.github/workflows/`:

### `deploy-hetzner.yml`
- Trigger: push to `main`
- Action: rsync plugins to Hetzner, run `upgrade.php`, purge caches.
- Paths filter excludes `_claude-skills-update/`, `product/`, etc.

### `test-hetzner.yml`
- Trigger: push to `main` affecting `playbooks/**`, `plugins/**`,
  `.github/workflows/test-hetzner.yml`, or the publish script itself.
- Concurrency group: `test-hetzner`, `cancel-in-progress: false` — so
  runs queue rather than replace each other.
- Jobs: `phpunit` and `behat`, each SSH-ing into Hetzner and invoking
  `lernhive-test` (the server-side wrapper for `playbooks/test.sh`).
- Both jobs `tee` test output to `/tmp/test-output-<job>.log` and then
  run `.github/scripts/publish-test-result.sh`, which pushes a machine-
  readable result bundle to the `ci/test-results` orphan branch.

### `ci/test-results` feedback loop

Why it exists: the Cowork sandbox egress allowlist only permits
`github.com` (git) and `dev.lernhive.de` — `api.github.com`,
`raw.githubusercontent.com`, and `codeload.github.com` are all blocked.
So Claude cannot call the GitHub API to read workflow logs.

Workaround: the workflow itself writes results to a dedicated orphan
branch (`ci/test-results`) which Claude can `git fetch` from the shadow
clone via the normal git protocol.

Files produced per job:
```
runs/
├── latest-phpunit.log        # full captured output of most recent run
├── latest-phpunit.json       # {status, exit_code, has_warnings, sha, tests, assertions, ...}
├── latest-behat.log
├── latest-behat.json
└── archive/
    ├── <ts>-<sha>-phpunit.log
    └── <ts>-<sha>-behat.log
```

Usage from Claude:
```bash
SHADOW=/sessions/<session>/shadow/lernhive
git -C "$SHADOW" fetch origin 'refs/heads/ci/test-results:refs/remotes/origin/ci/test-results'
git -C "$SHADOW" show origin/ci/test-results:runs/latest-phpunit.json
git -C "$SHADOW" show origin/ci/test-results:runs/latest-phpunit.log | tail -80
```

The publish script needs `permissions: contents: write` on the workflow,
and it uses the default `GITHUB_TOKEN` — no PAT required for publishing
inside the repo's own branches.

## 6. Shadow-clone + PAT git workflow (from Cowork sandbox)

Cowork mounts the user's git checkout at `/sessions/<session>/mnt/<repo>/`
but **does not let Claude `git push` directly from that mount**. Pattern:

1. Maintain a shadow clone at `/sessions/<session>/shadow/<repo>/`.
2. On each session, `git fetch --quiet origin main` + `git reset --hard`
   the shadow to origin/main.
3. `cp` the modified file(s) from the workspace mount into the shadow.
4. `git -c user.name=... -c user.email=... commit` inside the shadow.
5. `git push https://x-access-token:${PAT}@github.com/<org>/<repo>.git HEAD:main`.

The PAT + git identity live at
`/sessions/<session>/mnt/.auto-memory/cowork-secrets/`:
```
github.pat          # fine-grained PAT with content:write on the repo
git-identity        # name=...\nemail=...
```

These are created once by the user and persist across sessions. Never
echo the PAT in tool output. Unset it (`unset TOKEN`) after the push.

## 7. When something goes wrong

Most deploy pipeline issues fall into three buckets:

1. **Upgrade.php crashes** — read `references/05-errors.md` for the
   catalog of XMLDB/install.xml/version.php mistakes.
2. **Cache staleness** — always `purge_caches.php` after deploy; in dev
   disable OPcache timestamp validation or set
   `opcache.revalidate_freq=0`.
3. **Permissions** — Moodle runs as `www-data` inside the container;
   `docker cp` may leave files owned by `root`. Fix with
   `docker exec <container> chown -R www-data:www-data <path>` after
   the copy.

For the CI feedback loop specifically, if `ci/test-results` stops
updating check:
- Is the workflow actually running? Look at `git log` on main to see
  whether the push matched the workflow `paths:` filter.
- Is `concurrency: test-hetzner` blocked by an older run? Queued runs
  don't cancel each other (by design), so one stuck run blocks all
  follow-ups.
- Did `publish-test-result.sh` error out? The workflow logs would say
  so — but since those are API-gated, the reliable fallback is
  `ssh <server> 'journalctl -u ...' ` if you have shell access.
