# Caddy reverse proxy for LernHive

TLS termination for the Moodle stack running under `moodle-docker` on the
Hetzner server. Caddy sits in front of the `webserver` service, handles
Let's Encrypt certificate issuance and renewal automatically, and proxies
plain HTTP to the Moodle container over the internal Docker network.

## Files

| file                 | purpose                                        |
|----------------------|------------------------------------------------|
| `docker-compose.yml` | Caddy service definition                       |
| `Caddyfile`          | Reverse proxy config (env-var driven)          |
| `.env`               | Generated by `provision.sh` at install time    |
| `README.md`          | This file                                      |

## Required environment variables

Rendered into `.env` by `playbooks/provision.sh`:

```
LERNHIVE_DOMAIN=dev.lernhive.de
LETSENCRYPT_EMAIL=admin@example.com
MOODLE_NETWORK=lernhive_default
MOODLE_UPSTREAM=webserver:8000
```

- `LERNHIVE_DOMAIN` — the public FQDN Moodle will be reachable under.
  DNS A/AAAA record must already point to the server before first start,
  otherwise the Let's Encrypt challenge will fail.
- `LETSENCRYPT_EMAIL` — contact address for the ACME account. Only used
  for expiry warnings when renewal fails; must be valid.
- `MOODLE_NETWORK` — the existing moodle-docker Docker network to attach
  to. Detect with `docker network ls | grep moodle`.
- `MOODLE_UPSTREAM` — service name + internal port of the Moodle
  webserver container. For moodle-docker the default is
  `webserver:8000`.

## Manual operations

```bash
cd /opt/caddy

# Start / stop / reload
docker compose up -d
docker compose down
docker compose exec caddy caddy reload --config /etc/caddy/Caddyfile

# Follow logs (TLS issuance, access log)
docker compose logs -f caddy

# Force certificate re-issue (rare; usually not needed)
docker compose down
docker volume rm caddy_caddy_data
docker compose up -d
```

## Troubleshooting first cert issuance

1. DNS wrong? `dig +short A dev.lernhive.de` must return the server IP.
2. Port 80 blocked? UFW must allow 80/tcp — Let's Encrypt HTTP-01
   challenge uses port 80, even when the site runs on 443.
3. Let's Encrypt rate limit? Switch Caddyfile to staging ACME
   (`acme_ca https://acme-staging-v02.api.letsencrypt.org/directory`),
   restart, verify success, then swap back.
4. Another service on port 80? `lsof -iTCP:80 -sTCP:LISTEN` on the host,
   or `docker ps --format '{{.Names}} {{.Ports}}' | grep ':80'`.

## Security notes

- The Caddy container exposes **only** 80/443/443-udp on the host; the
  Moodle webserver container should no longer publish any host ports
  once Caddy is in place (update the moodle-docker override accordingly).
- HSTS is set to 180 days initially. Bump to 2 years
  (`max-age=63072000; includeSubDomains; preload`) only after you're
  confident the site will stay HTTPS permanently.
- Caddy runs as non-root inside the image. Data volumes (`caddy_data`,
  `caddy_config`) persist issued certificates and account keys — do not
  delete them casually or you'll force re-issuance.
